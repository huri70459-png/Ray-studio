## 4. Rationale

The project owner selects **D1 (Foundation First)** because it:

1. **Maximizes architectural correctness** — builds parse → extract → graph → index in technical dependency order rather than inventing sibling modules behind ports prematurely.  
2. **Preserves dependency integrity** — full-spec evidence chain is `103 → 104 → 105 → 102`; sequencing matches that chain for the Phase B.2 path.  
3. **Minimizes long-term technical debt** — reduces irreversible contract and ingestion-stack debt before locking an indexer surface on incomplete foundations.  
4. **Best aligns with the Engineering Constitution and Ray Studio’s graph-first architecture** — graph remains source of truth for rich entities; ingestion builds toward a durable graph surface; 101 B.1 ports remain valid without forcing live adapters first.  
5. **Keeps future layers explicitly deferred** — Memory Engine (201), Gateway, Providers, and unrelated modules remain outside Phase B.2 first-target scope until separately authorized.

## 5. Selected Strategy

**Name:** D1 — Foundation First  

**Intent:** Govern and later implement the code-ingestion stack from the parser foundation upward, with Module 105 constrained to a **Phase B.2 slice** (not full-spec Memory-backed graph storage), and with 101 live adapter integration and Memory Engine deferred to later authorized work.

**Primary optimization objective:** Minimize irreversible architectural debt in the code-ingestion stack (dependency integrity + contract durability).
